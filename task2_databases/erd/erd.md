# Entity-Relationship Diagram — `energy_ts`

**Author: Celine Shoga**

The schema is normalised around the observation timestamp. `reading` is the central
fact table (holds the target, appliance energy). Each reading has exactly one indoor
sensor record and one weather record, linked by `reading_id` (1-to-1). This keeps the
fact table narrow and lets the API/queries load only the tables they need.

```mermaid
erDiagram
    reading ||--|| indoor_climate : "has (1:1)"
    reading ||--|| weather        : "has (1:1)"

    reading {
        int      reading_id PK
        datetime reading_ts "UNIQUE, indexed"
        int      appliances_wh "target (Wh)"
        int      lights_wh
    }
    indoor_climate {
        int    reading_id PK,FK
        double t1
        double rh_1
        double t2
        double rh_2
        double t3_to_t9 "... T3..T9"
        double rh_3_to_rh_9 "... RH_3..RH_9"
    }
    weather {
        int    reading_id PK,FK
        double t_out
        double press_mm_hg
        double rh_out
        double windspeed
        double visibility
        double tdewpoint
    }
```
